# Правила проекта (FSD + RN + TS)

Краткие договорённости для ежедневной работы. Не handbook — просто ориентир.

## 1. Куда что класть (FSD)

| Слой         | Смысл                                                 | Примеры                                                       |
| ------------ | ----------------------------------------------------- | ------------------------------------------------------------- |
| **app**      | Сборка приложения: провайдеры, навигация, store, тема | `AppProviders`, корневой навигатор                            |
| **pages**    | Экраны: композиция виджетов и фич, мало логики        | один экран = одна папка/страница                              |
| **widgets**  | Крупные UI-блоки из нескольких сущностей/фич          | шапка экрана, карточка-агрегат                                |
| **features** | Пользовательские действия и сценарии                  | «добавить в избранное», фильтр, форма логина                  |
| **entities** | Бизнес-сущности: данные + мелкий UI вокруг них        | пользователь, продукт, типы/API сущности                      |
| **shared**   | Всё переиспользуемое без домена                       | `ui`, `lib`, `api` (клиент), `hooks`, `config`, `types` общие |

Слайсы внутри слоя — по смыслу одной темы (`features/auth-login`, `entities/product`).

## 2. Что **не** класть в `shared`

- Доменные модели и правила конкретного продукта.
- Компоненты/хуки, которые имеют смысл только в одном флоу (если позже понадобится ещё раз — вынесешь).
- «Универсальные» обёртки над одним вызовом API без второго реального потребителя.
- Константы и стили, которые относятся к одной фиче/сущности.

`shared` — для того, что уже **реально** повторяется между слайсами.

## 3. Импорты

- Слайс снаружи импортирует только через **публичный API** (`index.ts` в корне слайса). Внутри слайса — как удобно.
- **Направление зависимостей:** `app` → `pages` → `widgets` → `features` → `entities` → `shared`. Слой не импортирует то, что выше по иерархии.
- Предпочитай **относительные пути внутри слайса** и **алиас на слой/слайс** снаружи (как настроено в проекте) — главное, единообразие.
- Не тянуть из `shared` всё подряд в один файл: импортируй то, что нужно слайсу по смыслу.

## 4. Когда заводить `feature` / `entity` / `widget`

- **Entity** — есть устойчивая «вещь» предметной области, её типы/запросы/маленький презентационный кусок переиспользуются.
- **Feature** — есть законченное **действие пользователя** или шаг сценария (кнопка + логика + возможно локальный стейт).
- **Widget** — на экране нужен **составной** блок из нескольких entities/features, иначе страница раздувается.

Если сомневаешься: начни в `pages`, при втором использовании — вынеси в нужный слой.

## 5. Не делать «на будущее»

- Не плодить пустые папки, абстрактные `services`/`managers` без текущей задачи.
- Не выносить в `shared` до появления второго потребителя (кроме явно общих вещей типа кнопки/хука формата даты).
- Не проектировать generic-архитектуру под гипотетические экраны.

Делай минимум, который закрывает **текущий** экран или задачу.

## 6. Лишние абстракции

- Один уровень обёртки должен давать ясную выгоду (меньше дубля, проще тест, стабильный контракт).
- Если интерфейс/фабрика/базовый класс используется в одном месте — скорее всего это лишнее; слей с реализацией.
- Повтор кода 2 раза — ещё не повод для общей библиотеки; 3+ или явная боль поддержки — повод обсудить вынос.

## 7. Размер файлов и модулей

- Ориентир: файл, который не помещается на экран без постоянного скролла, — кандидат на разбиение по смыслу (не по «100 строк» механически).
- Несколько крупных хуков/компонентов в одном файле — разделяй, когда читать/править неудобно.
- Папка слайса должна отражать одну тему; если внутри десятки несвязанных файлов — возможно, это два слайса.

---

При конфликте «красиво по FSD» vs «быстро и понятно соло» — выбирай понятность кода; слои можно слегка подправить при следующем рефакторе.
